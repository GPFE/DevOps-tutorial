## Theory (20 min)

### GitHub Secrets explained

GitHub Secrets are **encrypted environment variables** stored in GitHub's vault. When a workflow runs:

1. GitHub decrypts the secret inside the Actions runner (an isolated VM)
2. The secret is injected as an environment variable
3. GitHub **masks** the value in logs — if your code accidentally prints it, GitHub replaces the output with `***`
4. The secret is never accessible to PR workflows from forked repositories (security protection against PRs from strangers exfiltrating your keys)

**Secrets are not magic** — they are just encrypted strings. Their security depends on:
- Keeping them out of your code (never hardcode an API key)
- Rotating them regularly
- Using the principle of least privilege (give a key only the permissions it needs)

---

### The secrets in your project

| Secret | What it is | What happens if it leaks |
|--------|-----------|--------------------------|
| `GITHUB_TOKEN` | Auto-generated by GitHub per workflow run | Expires after the run — low risk |
| `AZURE_FOUNDRY_RESOURCE` | The name of your Azure AI Foundry resource | Attacker knows *where* to call — but still needs the key |
| `AZURE_FOUNDRY_API_KEY` | API key for your Claude deployment on Azure | Attacker can call Claude at your expense — rotate immediately |

### What to do if a secret leaks

If you accidentally commit an API key to a public repo:
1. **Assume it is compromised** — GitHub's secret scanning bots and malicious actors scrape new commits in seconds
2. **Rotate it immediately** — go to [ai.azure.com](https://ai.azure.com) → your project **Home** page → regenerate the **Project API key**
3. **Update the GitHub Secret** with the new value
4. **Remove it from git history** — `git filter-repo` or GitHub's "Remove secret" tool
5. Check Azure **Cost Management** for unexpected charges

## Tasks

### 1. Audit your secrets

Go to **Settings → Secrets and variables → Actions** in your GitHub repository.

You should see:
- `AZURE_FOUNDRY_RESOURCE`
- `AZURE_FOUNDRY_API_KEY`

`GITHUB_TOKEN` is not listed there — it is automatically available in every workflow as `${{ secrets.GITHUB_TOKEN }}`.

For each secret, answer:
- What service does it belong to?
- What can an attacker do with it?
- Where would I go to rotate it?

Write the answers in a comment in the PR you open for this week (see Task 4).

### 2. Test secret masking in logs

Add a temporary step to any workflow, push, and look at the logs:

```yaml
- name: Test secret masking
  run: echo "The key is ${{ secrets.AZURE_FOUNDRY_API_KEY }}"
```

In the logs you will see:
```
The key is ***
```

Remove this step after testing — never intentionally log secrets.

### 3. Draw the full pipeline

On paper or in a diagram tool (even a comment in your PR), draw this pipeline and fill in every component:

```
Developer writes code
        ↓
git push (feature branch)
        ↓
GitHub receives the push
        ↓
[GitHub Actions - CI]
  ↓ htmlhint validates index.html
  ↓ PASS ✅ / FAIL ❌
        ↓
[Cloudflare Pages - Preview]
  ↓ Builds preview deployment
  ↓ Posts preview URL on PR
        ↓
[GitHub Actions - AI Review]
  ↓ Gets PR diff
  ↓ Calls Claude API (Azure AI Foundry)
  ↓ Posts review comment on PR
        ↓
Developer reviews feedback → fixes issues → pushes again
        ↓
All checks pass ✅
        ↓
PR merged to main
        ↓
[Cloudflare Pages - Production]
  ↓ Builds production deployment
  ↓ Live at https://your-project.pages.dev
```

For each box, note:
- What triggers it?
- What does it produce?
- What secret (if any) does it use?
